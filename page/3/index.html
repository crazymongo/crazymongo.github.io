Java-ByteArrayOutputStream

此类实现一个输出流，其中的数据被写入一个byte数据。缓冲区会随着数据的不断写入而自动增长。可使用toByteArray()和toString()获取数据。关闭ByteArrayOutputSream无效。此类中的方法在关闭此流后仍可被调用，而不会产生任何IOException。

此类的关闭函数定义如下：

 /**
 * Closing a <tt>ByteArrayOutputStream</tt> has no effect. The methods in
 * this class can be called after the stream has been closed without
 * generating an <tt>IOException</tt>.
 */
public void close() throws IOException {
}

Java-FilterOutputStream

此类是所有过滤输出流类的父类。这些流位于已存在的输出流（基础输出流）之上，他们将已存在的输出流作为其基本数据接收器，但可能直接传输数据或提供一些额外的功能。

FilterOutputStream类本身只是简单地重写那些将所有请求传递给所包含输出流OUtputStream的所有方法。FilterOutputStream的子类可以进一步的重写这些方法中的一些，并且还可以提供一些额外的方法和字段。

Java-Closeable-Flushable

接口Flushable是可刷新数据的目标地。调用flush方法将所有已缓冲数据输出写入底层流。其定义如下：

public interface Flushable {

    /**
     * Flushes this stream by writing any buffered output to the underlying
     * stream.
     *
     * @throws IOException If an I/O error occurs
     */
    void flush() throws IOException;
}

接口Closeable是可以关闭的数据源或目标。调用close方法可释放对象保存的资源。定义如下：

public interface Closeable extends AutoCloseable {

/**
 * Closes this stream and releases any system resources associated
 * with it. If the stream is already closed then invoking this
 * method has no effect.
 *
 * <p> As noted in {@link AutoCloseable#close()}, cases where the
 * close may fail require careful attention. It is strongly advised
 * to relinquish the underlying resources and to internally
 * <em>mark</em> the {@code Closeable} as closed, prior to throwing
 * the {@code IOException}.
 *
 * @throws IOException if an I/O error occurs
 */
public void close() throws IOException;

}

Java-DataOutput

DataOutput接口用于将Java基本类型转化为一系列字节，并将这些字节写入二进制流。同时还提供了一个将String转换成UTF-8修改版格式并写入所得到的系列字节的工具。对应此接口中写入的所有方法，如果由于某种原因无法写入某个字节，则抛出IOException。

Java-DataInput

DataInput接口用于从二进制流中读取字节，并根据所有Java基本类型数据进行重构。同时还提供根据UTF-8修改版格式的数据重构String的工具

对于此接口中的所有数据读取例程来说，如果在读取所需字节数之前已经到达文件末尾（end of file），则抛出EOFException。如果由于某种原因无法读取任何字节，而不是在读取所需数量的字节之前已到达文件末尾，则抛出IOException，而不是EOFException。需要特别指出的是，如果流已被关闭，则可能抛出IOException。

相关方法介绍：

void readFully(byte b[]) throws IOException

从输入流中读取一些字节，并将它们存储在缓冲区数组b中。读取的字节数等于b的长度。

在出现以下条件之一以前，此方法将一直阻塞：

  • 输入数据的b.length个字节是可用的，在这种情况下，正常返回。
  • 检测到文件末尾，在这种情况下，抛出EOFException。
  • 发生I/O错误，在这种情况下，抛出IOException。

int skipBytes(int n) throws IOException

视图在输入流中跳过数据的n个字节，并丢弃跳过的字节。不过，可以跳过更少的字节数，该字节数甚至可以为零。这可能由很多情况引起；在已经跳过n个字节前到达文件末尾只是其中的一种可能。

该方法返回实际跳过的字节数。

Android-Studio-Gradle-初识

使用Android Studio新建一个工程后，默认会生成两个build.gradle文件，一个位于工程根目录，一个位于App目录下。还有另外一个文件settings.gradle。

根目录下的脚本文件是针对module的全局配置，它的作用域包含的所有module是通过settings.gradle来配置的。

App文件夹就是一个module，如果在当前工程中添加了一个新的module，就需要在settings.gradle文件中包含这个新的module。

RESTful

REST(Representational State Transfer)，在设计分布式系统时，REST是一种构建风格。它不是一个标准，而是一系列的约束，例如：无状态，具有客户端/服务器的关系，以及统一的接口。

REST原理：

  1. 暴露容易理解目录结构的URI
  2. 使用JSON或者XML格式的传输数据对象和属性
  3. 用HTTP方法（例如：get，post，put等）
  4. 无状态，及服务器端不存在客户端上下文。

Android-Messenger分析

因为Messenger引用了一个Handler对象，因此其它进程可以使用该Messenger向其本身所引用的Handler对象发送消息。Messenger本身包含AIDL的实现，因此可以实现ICP。这是一种比较简单的IPC实现，Messenger在一个线程中将所有请求进行排队，因此service在同一时刻只能接受一个请求。如果你想service在同一时刻处理多个请求，可以尝试使用AIDL。

如果你需要在不同进程交互的接口，你可以使用Messenger为你的service创建那样的接口

下面是一个使用Messenger实现简单的IPC的例子：

public class AppService extends Service {

    private Messenger mMessenger=new Messenger(new IncomingHandler());

    @Override
    public void onCreate() {
        // TODO Auto-generated method stub
        super.onCreate();
    }

    @Override
    public void onDestroy() {
        // TODO Auto-generated method stub
        super.onDestroy();
    }

    @Override
    public IBinder onBind(Intent arg0) {
        // TODO Auto-generated method stub
        return mMessenger.getBinder();
    }


    class IncomingHandler extends Handler{

        @Override
        public void handleMessage(Message msg) {
            switch(msg.what){
            case AppConstants.MSG_SAY_HELLO:
                Toast.makeText(getApplicationContext(), "hello!", Toast.LENGTH_LONG).show();

                try {
                    msg.replyTo.send(Message.obtain(null, AppConstants.MSG_SAY_HELLO));
                } catch (RemoteException e) {
                    // TODO Auto-generated catch block
                    e.printStackTrace();
                }
                break;
            }
            msg.recycle();
        }
    }

}

public class MainActivity extends Activity implements OnClickListener{

    private Messenger mMessenger;
    private Messenger mService;
    private Button btn_ation;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        // TODO Auto-generated method stub
        super.onCreate(savedInstanceState);
        bindService(new Intent(this, AppService.class), conn, Context.BIND_AUTO_CREATE);
        mMessenger=new Messenger(new IncomingHandler());
        setContentView(R.layout.activity_main);
        btn_ation=(Button) findViewById(R.id.btn_action);
        btn_ation.setOnClickListener(this);
    }

    private ServiceConnection conn=new ServiceConnection() {

        @Override
        public void onServiceDisconnected(ComponentName name) {
            // TODO Auto-generated method stub

        }

        @Override
        public void onServiceConnected(ComponentName name, IBinder service) {
            // TODO Auto-generated method stub
            mService=new Messenger(service);
            btn_ation.setEnabled(true);
        }
    };

    @Override
    public void onClick(View v) {
        switch(v.getId()){
        case R.id.btn_action:
            try {
                Message msg=Message.obtain(new IncomingHandler(), AppConstants.MSG_SAY_HELLO);
                msg.replyTo=mMessenger;
                mService.send(msg);
            } catch (RemoteException e) {
                // TODO Auto-generated catch block
                e.printStackTrace();
                Toast.makeText(this, "消息发送失败", Toast.LENGTH_LONG).show();
            }
            break;
        }

    }

    @Override
    protected void onDestroy() {
        // TODO Auto-generated method stub
        super.onDestroy();
        unbindService(conn);
    }



    class IncomingHandler extends Handler{

        @Override
        public void handleMessage(Message msg) {
            switch(msg.what){
            case AppConstants.MSG_SAY_HELLO:
                btn_ation.setText("Success");
                break;
            }
            msg.recycle();
        }
    }


}

Compared to AIDL

When you need to perform IPC, using a Messenger for your interface is simpler than implementing it with AIDL, because Messenger queues all calls to the service, whereas, a pure AIDL interface sends simultaneous requests to the service, which must then handle multi-threading.

For most applications, the service doesn’t need to perform multi-threading, so using a Messenger allows the service to handle one call at a time. If it’s important that your service be multi-threaded, then you should use AIDL to define your interface.

Android-Looper

Looper类主要作用就是对消息队列进行轮询，从队列中取出消息，然后进行处理。缺省情况下，线程没有消息处理机制，下面是为线程添加消息处理机制的例子：

class LooperThread extends Thread{

    public Handler mHandler;

    public void run(){
        Looper.prepare();
        mHandler=new Handler(){
            public void handleMessage(Message msg){
                //process incoming messages here
            }
        }

        Looper.loop();
    }
}

当我们调用Looper.prepare()方法时，就为当前线程创建了一个Looper对象，该对象由ThreadLocal进行保存。ThreadLocal类保证每个线程都有自己的Looper对象，并且其它线程无法访问。新创建的Looper对象中包含一个MessageQueue，然后通过调用Looper.loop()方法，对该MessageQueue进行轮询操作。

我们可以参考：HandlerThread和IntentService这两个类，为一个线程加入消息处理机制。
